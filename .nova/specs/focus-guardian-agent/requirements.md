# 要件定義書

## プロジェクト説明（入力）
ユーザの目標実現を支援する監視AIエージェント「(仮称) Focus Guardian」。DevOps × AI Agent Hackathon 2026 提出作品（締切 2026/7/10）。

### 概要
資格勉強など集中したい作業中のユーザを定期的にカメラ画像で観察し、サボり行動（スマホいじり・居眠り等）を Gemini のマルチモーダル画像理解で検知。検知時は、ユーザがヒアリング時に悪習慣ごとに選んだ通知（BGM 再生 または 言葉での声かけ）をスマートスピーカー等から 1 回実行し、作業に引き戻す。

### 構成要素
1. 画像取得: PC Webカメラ（ブラウザが定期キャプチャして Cloud Run に POST）。サーバ側は pull 型の `CameraSource.get_latest_frame()` インターフェースに統一し、将来 Nest Cam (SDM API / WebRTC) アダプタへ差し替え可能にする
2. 目標ヒアリング bot: 固定フローの対話（自由入力と選択肢の組み合わせ）で目標・悪習慣・通知方法を収集し、Gemini で悪習慣を検知条件に構造化して Firestore に保存
3. 監視・介入エージェント: ADK + Gemini によるマルチエージェント構成（ヒアリング/監視判定/介入実行）。Cloud Run で動作。介入出力は `SpeakerAdapter.speak()` / `play_bgm()` インターフェースで抽象化し、AlexaVoiceMonkeyAdapter（今回実装、Voice Monkey API 経由）と GoogleCastAdapter（将来）を切り替え可能にする。ブラウザ再生もフォールバックとして持つ

### 技術スタック（ハッカソン要件準拠）
- Cloud Run（必須①）/ ADK + Gemini API（必須②）/ Firestore
- GitHub 公開リポジトリ + GitHub Actions CI/CD（「まわす」評価軸対応）
- 提出物: デプロイURL、Proto Pedia 登録（デモ動画・アーキテクチャ図・ストーリー）

### 設計上の重要事項
- 画像はプライバシー配慮のため判定後に破棄（保存しない）
- 審査基準「エージェントである必然性」対応: ①自由入力の悪習慣を画像判定可能な検知条件へ Gemini で構造化、②画像からの継続的な自律判定と介入タイミングの自己決定（継続時間・不在との区別・再介入抑制）を柱とする

## はじめに

Focus Guardian は、集中したい作業に取り組むユーザを支援する監視 AI エージェントである。固定フローの対話で目標・悪習慣・通知方法をヒアリングし、Web カメラの定期画像を Gemini のマルチモーダル理解で判定、悪習慣を検知したらユーザが事前に選んだ通知（BGM または 言葉）を 1 回実行する。本要件は「ヒアリング → 観察 → 判定 → 介入」のループを中核に、ハッカソン提出要件(Cloud Run 稼働、公開リポジトリ、CI/CD)までを対象範囲とする。

## 要件

### 要件1: 目標・悪習慣の固定フローヒアリング

**目的:** 集中したい作業があるユーザとして、迷わず答えられる決まった流れの対話で目標と悪習慣、通知方法を設定したい。それにより短時間で確実に監視設定を完了できる。

#### 受け入れ基準
1. When ヒアリングが開始された場合, the ヒアリング bot shall 「実現したい目標はありますか？」と質問し、自由入力で目標を受け付ける
2. When 目標が入力された場合, the ヒアリング bot shall 障壁となっている悪習慣を質問し、自由入力で受け付ける
3. When 悪習慣が入力された場合, the ヒアリング bot shall 検知時の通知方法を選択肢（BGM を流す / 言葉で通知する）として提示する
4. Where 通知方法として BGM が選択された場合, the ヒアリング bot shall BGM の種類をシステム提示の選択肢から選ばせる
5. Where 通知方法として言葉が選択された場合, the ヒアリング bot shall 通知フレーズ（例: 「起きてください」）を自由入力で受け付ける
6. When 1 つの悪習慣の通知設定が完了した場合, the ヒアリング bot shall 「他に悪習慣はありますか？」と質問し、追加の悪習慣の自由入力または「もう大丈夫」の選択肢を受け付ける
7. When 「もう大丈夫」が選択された場合, the ヒアリング bot shall 終了メッセージを提示し、監視設定を Firestore に保存する
8. When 悪習慣が入力された場合, the ヒアリングエージェント shall Gemini により自由入力の悪習慣を画像判定可能な検知条件に構造化する
9. The 監視設定 shall 目標と、悪習慣ごとの検知条件・通知方法（BGM の種類 または 通知フレーズ）を含む
10. If 定義された固定フロー以外の入力タイミングで想定外の応答があった場合, the ヒアリング bot shall 現在のステップの質問を再提示する

### 要件2: カメラ画像の定期取得

**目的:** ユーザとして、PC の Web カメラで作業中の様子を自動的に取得させたい。それにより操作の手間なく監視を受けられる。

#### 受け入れ基準
1. When 集中セッションが開始された場合, the フロントエンド shall Web カメラのアクセス許可を取得し、設定された間隔で静止画をキャプチャして監視 API に送信する
2. When 監視 API が画像を受信した場合, the 監視サービス shall 最新フレームとして保持し、判定処理から取得可能な状態にする
3. The 監視サービス shall カメラ入力源（Web カメラ / 将来の Nest Cam）を設定変更のみで差し替え可能な共通インターフェースで抽象化する
4. If カメラのアクセス許可が拒否された場合, the フロントエンド shall エラーメッセージと復旧手順を表示し、セッションを開始しない
5. If 画像送信に失敗した場合, the フロントエンド shall 再送信を試み、連続して失敗した場合はユーザに通知する

### 要件3: サボり行動の判定

**目的:** ユーザとして、設定した悪習慣を自動的に検知してほしい。それにより自覚のないまま集中が途切れることを防げる。

#### 受け入れ基準
1. When 判定周期が到来した場合, the 判定エージェント shall 最新の画像を取得し、Gemini のマルチモーダル画像理解により監視設定の検知条件に照らして行動を分類する
2. The 判定エージェント shall 判定結果（集中 / 悪習慣の種別 / 不在）と確信度を判定履歴として Firestore に記録する
3. When 悪習慣がシステム既定の許容閾値を超えて継続した場合, the 判定エージェント shall 介入エージェントに該当悪習慣の介入要求を発行する
4. If 画像から人物が検出できない場合, the 判定エージェント shall 「不在」として記録し、悪習慣としての介入を行わない
5. If Gemini API の呼び出しに失敗した場合, the 監視サービス shall エラーを記録し、次の判定周期で再試行する

### 要件4: 介入の実行

**目的:** ユーザとして、悪習慣が検知されたら自分が選んだ方法で 1 回だけ通知してほしい。それにより煩わしさなく作業に戻れる。

#### 受け入れ基準
1. When 介入要求を受信した場合, the 介入エージェント shall 該当する悪習慣に設定された通知方法（BGM 再生 または 言葉での通知）を 1 回実行する
2. Where 通知方法が BGM の場合, the システム shall ヒアリングで選択された BGM を再生する
3. Where 通知方法が言葉の場合, the システム shall ヒアリングで設定されたフレーズを音声で通知する
4. While 同一の悪習慣が継続して検知されている間, the 介入エージェント shall 再介入を行わない
5. When 集中状態への復帰後に同じ悪習慣が再び検知された場合, the 介入エージェント shall 改めて 1 回介入を実行する
6. When 介入手段が実行される場合, the システム shall 出力デバイス共通インターフェースを通じて介入を実行する
7. Where Alexa が出力先として設定されている場合, the システム shall Voice Monkey API 経由で BGM 再生・言葉での通知を実行する
8. If 外部スピーカーへの出力に失敗した場合, the システム shall ブラウザでの音声再生にフォールバックする
9. The システム shall 出力デバイス（Alexa / 将来の Google Home）を設定変更のみで切り替え可能な構造とする
10. When 介入を実行した場合, the システム shall 介入した悪習慣・通知方法・時刻を介入履歴として Firestore に記録する

### 要件5: 介入結果の記録

**目的:** ユーザとして、介入後に作業へ戻れたかを記録してほしい。それによりセッション後に自分の傾向を振り返られる。

#### 受け入れ基準
1. When 介入実行後の判定周期が到来した場合, the 介入エージェント shall 直後の判定結果に基づき介入結果（作業に復帰したか）を評価する
2. When 介入結果が評価された場合, the システム shall 評価結果を介入履歴に記録する
3. While ユーザの集中状態が継続している間, the 介入エージェント shall 介入を実行しない

### 要件6: セッション管理と可視化

**目的:** ユーザとして、セッションの開始・終了を簡単に操作し、結果を振り返りたい。それにより日々の集中の質を把握できる。

#### 受け入れ基準
1. When ユーザがセッション開始を指示した場合, the システム shall 保存済みの監視設定を読み込み、監視ループを開始する
2. While セッションが進行中の間, the ダッシュボード shall 現在の判定状態・経過時間・介入回数を表示する
3. When ユーザがセッション終了を指示した場合, the システム shall 画像の取得を停止し、セッションサマリー（集中時間、悪習慣の検知回数、介入回数と結果)を表示する
4. The フロントエンド shall 監視設定済みのユーザがセッション開始まで 3 操作以内で到達できる導線を備える

### 要件7: プライバシー保護とセキュリティ

**目的:** ユーザとして、カメラ画像が保存・流出しない保証がほしい。それにより安心して監視機能を使える。

#### 受け入れ基準
1. When 画像の判定が完了した場合, the システム shall 判定に使用した画像を破棄し、永続化しない
2. The システム shall 永続化するデータを構造化された監視設定・判定結果・介入履歴に限定する
3. While カメラがキャプチャ中の間, the フロントエンド shall 撮影中であることを画面上に常時明示する
4. The システム shall API キー等のシークレットをソースコードに含めず、環境変数または Secret Manager で管理する

### 要件8: 運用・デプロイ（ハッカソン提出要件）

**目的:** 提出者として、ハッカソンの技術要件と提出要件を満たす形で本番稼働させたい。それにより審査対象として成立させる。

#### 受け入れ基準
1. The バックエンド shall Cloud Run 上で動作する
2. The ソースコード shall GitHub の公開リポジトリで管理される
3. When main ブランチに変更がマージされた場合, the CI/CD パイプライン shall テストの実行と Cloud Run へのデプロイを自動的に行う
4. The デプロイ環境 shall 審査期間終了（2026/7/30）まで動作確認可能な状態を維持する
