# 実装計画

- [ ] 1. プロジェクト基盤とリスク潰し
- [ ] 1.1 プロジェクト雛形と開発環境の構築
  - バックエンド（FastAPI + ADK 同居）とフロントエンド（Vite + React + TypeScript）のモノレポ雛形を作成し、ローカルで両者が起動して疎通することを確認する
  - Firestore エミュレータを含むローカル開発手順を整備する
  - GitHub 公開リポジトリを作成して初期コミットを行う
  - _Requirements: 8.1, 8.2_

- [ ] 1.2 (P) 外部依存の縦串検証スパイク
  - ADK v2.3 の LlmAgent 単発実行 + output_schema で構造化出力が返ることを実機確認する（schema 無視の報告があるため instruction にも形式を明記して検証）
  - Voice Monkey v3 の announce API で公開 MP3 URL が Echo 実機で再生されることを確認する
  - Cloud Text-to-Speech で日本語フレーズの MP3 が生成できることを確認する
  - 検証結果（動いた設定値）を記録し、以降のタスクの前提とする
  - 1.1 完了後に 1.3 と並行実行可能
  - _Requirements: 3.1, 4.3, 4.7_

- [ ] 1.3 (P) CI/CD パイプラインとシークレット管理
  - GitHub Actions で main マージ時にテスト実行 → Cloud Run デプロイまで自動化する（Workload Identity Federation 認証）
  - Voice Monkey トークンを Secret Manager 経由で Cloud Run に注入し、ソースコードにシークレットを含めない構成にする
  - Cloud Run の稼働確認（ヘルスチェックエンドポイント）と max-instances=1 の設定を行う
  - 1.1 完了後に 1.2 と並行実行可能
  - _Requirements: 7.4, 8.1, 8.3, 8.4_

- [ ] 2. Firestore リポジトリとドメインモデル
  - 監視設定・ヒアリング進行状態・セッション・判定履歴・介入履歴の Pydantic モデルと Firestore リポジトリを実装する
  - 永続化対象を構造化データのみに限定し、画像・生体情報を含むフィールドが存在しないことをモデルで保証する
  - セッション本体の read-modify-write をトランザクションで保護する
  - _Requirements: 1.9, 7.2_

- [ ] 3. ヒアリング機能
- [ ] 3.1 固定フローステートマシンとヒアリング API
  - 目標（自由入力）→ 悪習慣（自由入力）→ 通知方法（選択肢）→ BGM 種類（選択肢）/ フレーズ（自由入力）→ 追加悪習慣 or「もう大丈夫」→ 終了メッセージ、の全遷移を実装する
  - 各ステップの bot メッセージ・入力モード・選択肢を API レスポンスで返す
  - 想定外の入力（選択肢ステップへの自由入力等）では状態を変えず同一質問を再提示する
  - 完了時に監視設定を保存し、設定内容の要約を終了メッセージとして提示する
  - 進行状態は Firestore に保存し、全遷移をユニットテストで網羅する
  - 検知条件の構造化（3.2）はスタブで開始可能
  - _Requirements: 1.1, 1.2, 1.3, 1.4, 1.5, 1.6, 1.7, 1.9, 1.10_

- [ ] 3.2 (P) 悪習慣の検知条件構造化エージェント
  - 自由入力の悪習慣テキストから、画像判定用の視覚的手がかりと判定ヒント文を生成する ADK エージェントを実装する（構造化出力専用・単発実行）
  - 構造化失敗時は例外とし、ヒアリング側が同一質問を再提示できるようにする
  - _Requirements: 1.8_

- [ ] 3.3 (P) 通知音声アセットの準備と生成
  - BGM の固定ラインアップ（数種類）を用意し、Cloud Storage の公開バケットに配置して選択肢 ID と URL を対応付ける
  - ヒアリング完了時にフレーズを Cloud Text-to-Speech で日本語 MP3 化し、公開 URL を監視設定に保存する
  - オブジェクト名は推測困難にする
  - _Requirements: 4.2, 4.3_

- [ ] 3.4 (P) ヒアリングチャット UI
  - API のレスポンスをそのまま描画するチャット画面（bot 吹き出し + 自由入力欄 / 選択肢ボタンの出し分け）を実装する
  - フロント側にフロー知識を持たせず、要件の会話例どおりの体験を再現する
  - _Requirements: 1.1, 1.3, 1.6, 1.10_

- [ ] 4. 監視・判定パイプライン
- [ ] 4.1 (P) フレーム受信 API とカメラ入力の抽象化
  - カメラ入力源の共通インターフェース（pull 型の最新フレーム取得）と、ブラウザ POST を受ける Web カメラ実装を作る
  - フレームはリクエストスコープでのみ保持し、処理完了後に破棄されること（永続化・ログ出力なし）を保証する
  - Content-Type・サイズ上限・JPEG 形式のバリデーションと、セッションあたりの簡易レート制限を実装する
  - _Requirements: 2.2, 2.3, 7.1_

- [ ] 4.2 (P) 画像判定エージェント
  - 監視設定の検知条件を組み込んだプロンプトで、画像を「集中 / 悪習慣（該当 ID）/ 不在」に分類する ADK エージェントを実装する（enum 構造化出力・単発実行・gemini-2.5-flash-lite）
  - few-shot 例と「不在 vs 画角外」「該当なし → 集中」の判定基準をプロンプトに明文化する
  - 出力の Pydantic バリデーションを行い、逸脱は例外として扱う
  - _Requirements: 3.1, 3.4_

- [ ] 4.3 判定サイクルの統括
  - フレーム受信 → 判定 → 履歴記録 → 介入判断 → レスポンス返却の 1 サイクルを統合する
  - 判定結果（状態・確信度・根拠）を判定履歴として記録する
  - Gemini 呼び出し失敗時はエラーとして記録して介入判断をスキップし、次のフレームで自然に再試行される挙動にする
  - 4.1 と 4.2 の完了が前提
  - _Requirements: 3.1, 3.2, 3.5_

- [ ] 5. 介入機能
- [ ] 5.1 介入ポリシー（発火判断・再介入抑制・結果評価）
  - 同一悪習慣の有効判定（確信度 0.7 以上）が連続 2 回で介入要求を発火する判断ロジックを実装する
  - 発火後は同一悪習慣の継続中は再介入せず、集中復帰後の再検知では改めて 1 回介入する状態遷移（armed / fired）を実装する
  - 不在判定は連続カウントをリセットし介入を発火させない
  - 介入実行時に悪習慣・通知方法・時刻を介入履歴に記録する
  - 介入直後の判定で作業復帰を評価し（復帰 / 未復帰）、介入履歴を更新する
  - 状態はすべてセッションドキュメントに永続化し、判断ロジックをユニットテストで網羅する
  - _Requirements: 3.3, 3.4, 4.1, 4.4, 4.5, 4.10, 5.1, 5.2, 5.3_

- [ ] 5.2 (P) スピーカー出力の抽象化と Voice Monkey 実装
  - 出力デバイスの共通インターフェースを定義し、環境変数で実装を切り替えられるようにする（Voice Monkey / ブラウザのみ運用）
  - Voice Monkey v3 の announce API で、設定された BGM またはフレーズ MP3 の URL を再生する実装を作る（Bearer トークン認証）
  - 429 応答は短いバックオフで 1 回だけ再試行し、失敗は結果として呼び出し元に返す
  - _Requirements: 4.2, 4.3, 4.6, 4.7, 4.9_

- [ ] 5.3 ブラウザフォールバック再生
  - スピーカー出力失敗時、フレーム応答に含まれる介入指示をフロントエンドが受け取り音声を再生する経路を実装する
  - セッション開始時の操作で AudioContext を有効化し、ブラウザの自動再生制限を回避する
  - 5.1 と 5.2 の完了が前提
  - _Requirements: 4.8_

- [ ] 6. セッション管理とフロントエンド
- [ ] 6.1 セッション管理 API
  - 監視設定を読み込んでセッションを開始し、終了時にサマリー（集中時間・悪習慣の検知回数・介入回数と結果）を集計して返す API を実装する
  - セッション ID は推測困難な UUID とし、終了済みセッションへのフレーム送信は拒否する
  - _Requirements: 6.1, 6.3_

- [ ] 6.2 (P) Web カメラキャプチャコントローラ
  - カメラ許可を取得し、設定間隔（既定 60 秒）で静止画をキャプチャ → 長辺 768px にリサイズ → JPEG 圧縮 → フレーム API へ送信する
  - 許可拒否時はエラーメッセージと復旧手順を表示してセッションを開始しない
  - 送信失敗時はリトライし、連続 3 回失敗でユーザに通知する
  - キャプチャ中インジケータを画面上に常時表示する
  - _Requirements: 2.1, 2.4, 2.5, 7.3_

- [ ] 6.3 (P) セッションダッシュボード UI
  - 現在の判定状態・経過時間・介入回数をフレーム応答から逐次更新して表示する
  - セッション終了ボタンとサマリー表示画面を実装する
  - 設定済みユーザがトップからセッション開始まで 3 操作以内で到達できる導線にする
  - _Requirements: 6.2, 6.3, 6.4_

- [ ] 7. 統合と検証
- [ ] 7.1 バックエンド統合テスト
  - ヒアリング開始 → 完了 → 設定保存 → セッション開始 → フレーム判定 → 介入 → 結果評価 → セッション終了の縦串を、外部依存（Gemini / Voice Monkey）のフェイク実装と Firestore エミュレータで検証する
  - スピーカー失敗時のブラウザフォールバック分岐と、Gemini 失敗時の再試行経路を含める
  - _Requirements: 3.5, 4.8, 5.1, 6.3_

- [ ]* 7.2 フロントエンド E2E テスト
  - ヒアリング完走とセッション開始〜ダッシュボード表示の主要導線を Playwright で自動化する（1.1〜1.7、2.1、6.2、6.4 の受け入れ基準の UI 面カバレッジ。バックエンド統合テストで機能検証済みのため MVP 後に延期可能）
  - _Requirements: 1.1, 2.1, 6.2, 6.4_
